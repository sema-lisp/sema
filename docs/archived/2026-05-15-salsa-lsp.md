# Plan: Salsa-style query graph in `sema-lsp`

## Verdict

**Don't do it. Not now. Confidence: high (~85%).**

The conversational pitch — "every LSP feature recomputes overlapping intermediate results, Salsa would share work" — does not survive contact with `crates/sema-lsp/src/lib.rs`. The backend **already** implements the exact caches a Salsa graph would materialize, by hand, with two strategically placed HashMaps and one `mtime`-keyed re-read. The end-to-end work the LSP does on a keystroke is **one parse + one `ScopeTree::build`**, and every interactive feature reads from that cached pair. Adopting Salsa would replace ~60 lines of explicit cache code with a framework that brings: a major API churn risk, an Rc-vs-Arc collision with `sema-core::Value`, async/non-Send complications with the tokio runtime, and a new debugging surface for a problem that does not exist in the profiler.

The strongest counterargument: if Sema's LSP is going to grow toward cross-file type inference, macro expansion as a query, or a project-wide def-use index of the kind rust-analyzer maintains, then Salsa earns its keep. But none of that is on the current roadmap (see `docs/adr.md`, `docs/IDEAS.md`), and Sema's semantics (dynamic Lisp, no types, late binding) cap the upside.

Recommendation: **target the actual gaps with surgical changes** (a real LRU on `import_cache`, eviction policy fix, scope tree only for the changed file, optional parallel `handle_workspace_symbols` via rayon) rather than a framework adoption.

## Current state — verified

All citations point at `crates/sema-lsp/src/lib.rs` unless noted.

### Architecture is already single-writer + cache-on-write

- `Backend` (tower-lsp side, async) is a thin dispatcher. Every request becomes an `LspRequest` enum value sent over an `UnboundedSender` (lib.rs:2789) and awaited on a `oneshot::channel` (e.g. lib.rs:2923-2933 for completion).
- `BackendState` lives on a dedicated `std::thread` spawned in `run_server` (lib.rs:3182). It owns all the `Rc`-bearing state; `Value` and `SpanMap` cannot cross threads (Value uses `Rc`, confirmed at `crates/sema-core/src/value.rs:565-595`).
- The thread loop (lib.rs:3189-3438) is a classic actor: blocking recv, batch consecutive `DocumentChanged` for the same URI (lib.rs:3217-3238), then dispatch.

### Parses ARE already cached — globally and per-import

- `BackendState::cached_parses: HashMap<String, CachedParse>` (lib.rs:1317). Updated on every `DocumentChanged` (lib.rs:3262-3271).
- `CachedParse` stores `ast`, `span_map`, `symbol_spans`, **and** `scope_tree` (lib.rs:1265-1270). Note: the scope tree is **already a derived query** materialized eagerly on every edit (lib.rs:3262).
- `BackendState::import_cache: HashMap<PathBuf, ImportCache>` (lib.rs:1315), keyed by absolute path with `mtime`-based invalidation (lib.rs:1361-1401). This is a hand-rolled Salsa-lite for cross-file lookups, with a crude eviction at 500 entries (lib.rs:1358, 1373-1383).
- `cached_user_defs` survives parse errors so completions don't blink while typing (lib.rs:1311, 3253-3260) — a deliberate "previous good value" pattern Salsa does not give you for free.

### Workspace scan is incremental and lazy already

- `WorkspaceScanner` (lib.rs:1091-1147) is a stack-based, directory-at-a-time DFS. It yields between dirs (lib.rs:3402-3434) and processes only `BATCH_SIZE = 10` files per turn (lib.rs:3410), explicitly to "yield to interactive requests between directories" (lib.rs:3403-3405).
- Re-enqueueing itself on the `deferred` VecDeque (lib.rs:3186, 3407, 3432-3434) implements cooperative scheduling against incoming user requests — including a try_recv preempt for new interactive work (lib.rs:3203-3208).

### Each feature reads from cache, does NOT re-parse

Every `handle_*` method takes `cached_parses.get(uri_str)` as its entry point:

- `handle_complete` (lib.rs:1403) — reads `cached_user_defs` and `cached_parses` scope tree.
- `handle_goto_definition` (lib.rs:1528) — reads `cached_parses` + scope tree + `import_cache` on miss.
- `handle_hover` (lib.rs:1771), `handle_references` (lib.rs:1600), `handle_document_highlight` (lib.rs:1704), `handle_rename` (lib.rs:2654), `handle_prepare_rename` (lib.rs:2614), `handle_signature_help` (lib.rs:2136), `handle_semantic_tokens_full` (lib.rs:2224), `handle_folding_ranges` (lib.rs:2340), `handle_inlay_hints` (lib.rs:2379), `handle_document_symbols` (lib.rs:2056), `handle_workspace_symbols` (lib.rs:2067), `handle_code_lens` (lib.rs:1489) — all pull from `cached_parses`.
- The only "re-parse on miss" code paths are `handle_code_lens` (lib.rs:1501) and `handle_execute_command` (lib.rs:1916), both annotated as "should be rare for open documents".

### Where redundancy *actually* exists (modest)

These are the real recomputations on each interactive request — none are individually expensive:

1. `user_definitions_from_ast(&ast, &span_map, &symbol_spans)` is called in `handle_goto_definition` (lib.rs:1583), `handle_hover` (lib.rs:1799, 1853), `handle_document_symbols` indirectly via `document_symbols_from_ast`, and `handle_workspace_symbols` (lib.rs:2080). It walks the AST top-level forms each time. Cost: O(top-level forms). Could be cached as a query.
2. `import_paths_from_ast` (helpers.rs:367) is called in `handle_goto_definition`, `handle_hover`, `handle_signature_help`, `handle_inlay_hints`, `handle_rename`. Same shape: top-level walk.
3. `document_symbols_from_ast` is recomputed on every `handle_document_symbols` and every `handle_workspace_symbols` call, even when AST is unchanged (lib.rs:2062, 2080, 2110).
4. `semantic_tokens_full`'s "collect user-defined function/macro names" pass (lib.rs:2228-2262) is redone on every call.

### The reader and AST shape

- `sema_reader::read_many_with_spans_recover(text)` (`crates/sema-reader/src/reader.rs:635`) returns `(Vec<Value>, SpanMap, Vec<(String, Span)>, Vec<SemaError>)` — error-recovering, which is why diagnostics arrive even on broken input (lib.rs:3241-3247).
- `SpanMap = HashMap<usize, Span>` keyed by **`Rc<Vec<Value>>` raw pointer address** (`crates/sema-core/src/error.rs:130`, used at `scope.rs:51-55`). This means the SpanMap is tightly coupled to the *exact* Rc allocations the reader produced. Any Salsa interning step that creates *new* nodes would invalidate the entire SpanMap. **This is the biggest single-fact blocker.**

## What a Salsa graph would look like (if you did it)

For completeness, the queries that fall out of the current code:

| Query | Input(s) | Output | Notes |
| --- | --- | --- | --- |
| `source_text(uri)` | input | `Arc<String>` | Salsa input slot |
| `parse(uri)` | `source_text` | `Arc<ParseResult>` = `(ast, span_map, symbol_spans, errors)` | Today: `read_many_with_spans_recover` |
| `scope_tree(uri)` | `parse(uri)` | `Arc<ScopeTree>` | Today: built eagerly on DocumentChanged |
| `user_defs(uri)` | `parse(uri)` | `Arc<Vec<(String, Range)>>` | Currently recomputed per request |
| `import_paths(uri)` | `parse(uri)` | `Arc<Vec<String>>` | Currently recomputed per request |
| `resolved_imports(uri)` | `import_paths(uri)` + URI base | `Arc<Vec<PathBuf>>` | Currently recomputed per request |
| `parse_import(path)` | `mtime(path)` + `source_text_of_path(path)` | `Arc<ParseResult>` | Today: `import_cache` |
| `document_symbols(uri)` | `parse(uri)` | `Arc<Vec<DocumentSymbol>>` | Currently recomputed per request |
| `semantic_tokens(uri)` | `parse(uri)` + `scope_tree(uri)` + `builtin_names` | `Arc<SemanticTokens>` | Currently recomputed per request |
| `goto_def(uri, pos)` | `parse(uri)`, `scope_tree(uri)`, `resolved_imports(uri)`, `parse_import(path)` | `Option<Location>` | Position-keyed — bad Salsa input (cardinality explosion) |

The position-keyed queries (`goto_def`, `hover`, `references`, `document_highlight`, `signature_help`) **should not** be Salsa queries — their input space is unbounded and they shouldn't be memoized. They should remain plain methods that consume Salsa-derived intermediate queries.

So in practice you'd Salsa-ify roughly six derived queries, behind one "database" object, with a manual driver loop on top. That ratio of ceremony-to-benefit is poor.

## Migration path (if forced)

Phased, but every phase is large:

**Phase 0 — Pre-work (3 days).** Make `Value` and `SpanMap` `Send + Sync` for queries (it cannot today; Decision: not happening). Workaround: keep all Salsa state on the backend thread, mark the DB with `!Send`. This means losing one of Salsa's selling points (parallelism). **Salsa 0.18+ requires `Send + Sync` on tracked values by default**; you would have to wrap every AST node in a newtype with manual unsafe impls, or fork. Verdict: deal-breaker as currently designed.

**Phase 1 — Introduce the database, mirror existing queries (5–7 days).** Wrap `cached_parses` and `import_cache` behind `#[salsa::input]` and `#[salsa::tracked]`. Keep all `handle_*` methods, just have them resolve through the DB. Zero behavior change. This is the only phase where rollback is cheap.

**Phase 2 — Migrate derived queries (3–5 days).** `user_defs`, `import_paths`, `document_symbols`, `semantic_tokens` become tracked. Drop their inline recomputation. This is where you start to see the (marginal) wins.

**Phase 3 — Move builtin names + builtin docs into the DB (2 days).** They're currently `HashMap`s computed once at startup; trivial to Salsa-ify.

**Phase 4 — Consider parallelism (open-ended).** Only meaningful if Phase 0's `Send + Sync` problem is solved. Not realistic without changing `sema-core`.

The work can be incremental file-by-file in theory, but the DB type is monolithic — you can't have "half the state inside Salsa, half outside" without dual writes. So Phase 1 is effectively big-bang.

## Blast radius

- **`sema-lsp` only** for Phase 1–3. No other crate needs to change.
- **`sema-core` would need changes** for Phase 4 (Send/Sync on `Value`, `SpanMap`). That is a massive ripple touching the evaluator, the VM, the stdlib — every crate. This is the same reason Decision #54 ("Async is VM-only, VM is default backend") and the existing `Rc`-everywhere choice exist. **The architecture is intentionally single-threaded for the language runtime.**
- **No public API breaks** in Phase 1–3. `pub use` re-exports from `lib.rs:17-23` and the helper module stay as-is.

## Risks

1. **Salsa API churn.** As of 2026-05, the published `salsa` crate (the rewrite, called salsa-2022 in the tracking issue, now shipped as `salsa` 0.x with attribute macros `#[salsa::input]`, `#[salsa::tracked]`, `#[salsa::accumulator]`) is the third major API generation in five years. rust-analyzer is on its own fork (`salsa` from the rust-analyzer repo) for exactly this reason. Pinning to an upstream version is a known maintenance tax. **Open question for the user: which Salsa do you intend? Upstream 0.x or the ra-fork?**
2. **`Rc` everywhere collision (critical).** `sema-core::Value` uses `Rc` for all heap variants (value.rs:570-594). `SpanMap` keys on `Rc::as_ptr` (scope.rs:51-55). Salsa's tracked values need `Send + Sync`. You cannot put `Vec<Value>` into a Salsa query as-is. Workarounds: (a) keep the DB `!Send` and lose parallelism; (b) introduce an `Arc`-based mirror type just for the LSP — doubles memory and adds a conversion hop on every parse.
3. **`SpanMap` is pointer-keyed.** Any cloning or re-interning step inside Salsa would invalidate every span lookup downstream. The reader's contract is "the `Rc<Vec<Value>>` you got is the one in the SpanMap." Wrapping in Salsa requires either keeping the Rc alive verbatim (defeats Salsa's value semantics) or migrating to an index-based span model (large refactor).
4. **tower-lsp async interaction.** The current design routes async requests through a sync actor specifically because `Value: !Send`. Salsa's parallel features want to call queries from multiple tokio tasks. If you keep the actor model, you neutralize most of Salsa's parallelism story.
5. **Debugging.** Today: stack trace points at `handle_goto_definition`. With Salsa: trace points at framework internals, query revisions, and durabilities. For a 3,400-line LSP, that is a regression in maintainability.
6. **The benefit is unmeasured.** No benchmarks in the repo (`benchmarks/` directory exists but contains nothing about LSP latency). The hypothesis "LSP is slow because of redundancy" is not supported by data. Before doing anything, measure.

## Decision dependencies

- **Decision #55 (open-upvalue model, PROPOSED).** Unrelated to LSP. No interaction.
- **Decision #56 (bytecode verifier, PROPOSED).** Unrelated to LSP. No interaction.
- **Decision #57 (span propagation through runtime errors, PROPOSED).** *Mild interaction.* #57 wants to thread `Span` from the reader through eval/VM error paths. It would NOT change `SpanMap`'s shape; it would attach `Option<Span>` to `SemaError` at the error site. **No conflict with this plan**, but if both land, the LSP can light up runtime errors (from `sema.runTopLevel`) with precise ranges — which is independent of Salsa.
- **Decision #58 (thread-local stdout hook, PROPOSED).** Unrelated.
- **Decision #54 (Async is VM-only, VM is default backend).** *Indirect.* Reaffirms the single-thread-per-runtime model. Reinforces why Salsa's parallelism story is wasted here.

## What you would NOT migrate

- **Position-keyed handlers** (`goto_def`, `hover`, `references`, `document_highlight`, `signature_help`, `prepare_rename`, `inlay_hint` at a point): unbounded input space. Keep as plain methods.
- **`handle_execute_command`** (lib.rs:1883): spawns a subprocess. Salsa adds nothing.
- **Workspace scanner state** (lib.rs:1091): a cooperative driver, not a function.
- **Diagnostic publishing** (lib.rs:3274-3277): a side-effect, not a query.
- **`builtin_docs`** (lib.rs:1313): computed once at startup, never invalidated.
- **`semantic_token_legend`** (lib.rs:1288): pure const-fn-equivalent.

## Recommendation

1. **First**: add a microbenchmark harness for LSP keystroke latency. The whole question turns on data we don't have.
2. **If hot path is parsing**: nothing to fix, it's already cached.
3. **If hot path is the redundant `user_defs`/`import_paths`/`document_symbols` walks**: add a `parse_revision: u64` on `CachedParse`, memoize derived results keyed by `(uri, revision)`. ~150 lines.
4. **If hot path is `handle_workspace_symbols` across hundreds of imports**: parallelize with rayon over `import_cache` entries; this is the one place where parallelism pays.
5. **Revisit Salsa only if** Sema grows real cross-file type inference, or macro expansion becomes a heavy query, or a `find_references` over thousands of files becomes a feature.

---

## Open questions

1. **Is the LSP actually slow today?** No benchmarks exist; no profiler trace in `docs/`. The whole motivation hangs on this.
2. **Which Salsa?** Upstream `salsa` 0.x or the rust-analyzer fork? They diverged years ago and the answer materially changes Phase 0.
3. **Future of `sema-core::Value`'s `Rc`-vs-`Arc` choice.** If a multi-threaded runtime is on the roadmap (not visible in PLAN.md), an `Arc` migration would unblock Salsa's parallelism story. If not, the architecture deliberately precludes Salsa's biggest win.
