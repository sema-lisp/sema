# MCP Bundle (.mcpb) — single cross-platform bundle from the release

**Status:** archived — implemented and verified against real release assets.
Manifest, Linux shim, pack script, jake recipes, and the release-time workflow
(`.github/workflows/mcpb.yml`) are committed. `jake mcpb.pack` was run against
the live v1.30.0 release: downloaded all 5 target archives, lipo'd a genuine
universal macOS binary (`x86_64 arm64`), and packed `dist/sema.mcpb`
(55 MB / 135 MB unpacked). Untried only inside the workflow itself — first live
CI run is planned for the release after the unified-scheduler work lands. The
registry entry is **live**: `com.sema-lang/sema` v1.30.0, DNS-verified (below).

## Decision

Ship **one** `.mcpb` that covers macOS, Linux, and Windows, built from the
per-target binaries cargo-dist already attaches to each GitHub release. Do it
**in the mono's release flow**, not a separate repo: the trigger is the mono's
release and the inputs are the mono's binaries, so a cross-repo dispatch would
only add a PAT and a silent-lag failure mode. MCPB packaging is release/CI
tooling (like cargo-dist itself), so it's clear of the "no dev-tool in shipped
runtime code" rule in AGENTS.md.

Distribute via **MCPB from the GitHub release** rather than cargo/crates.io.
That also simplifies registry ownership: an MCPB package is verified by its
GitHub-release asset URL, so we publish under `io.github.sema-lisp/sema` via
GitHub OAuth — **no DNS TXT record and no crates.io README marker needed**.

## The one real constraint: MCPB has no arch dimension

`mcp_config.platform_overrides` keys on OS only (`darwin` / `win32` / `linux`),
and there is no `${arch}` substitution variable. Our release builds 5 targets
(`dist-workspace.toml`): mac arm64+x64, linux arm64+x64, windows x64. To collapse
that into one bundle where each OS resolves to a single command:

- **macOS** — `lipo` the two arch binaries into one **universal2** binary →
  `darwin` override points straight at it.
- **Linux** — ship both arch binaries + a committed POSIX shim
  (`mcpb/sema-linux`) that `exec`s the right one by `uname -m`. This is why the
  Linux `command` is the shim, not a binary.
- **Windows** — x86_64 only.

## Files (committed)

- `mcpb/manifest.json` — bundle manifest, `version` is `__VERSION__` (substituted
  at pack time). `server.type: binary`, entry `server/bin/sema-linux`, with
  `platform_overrides` for `darwin`/`win32`. Validated by `mcpb validate`.
- `mcpb/sema-linux` — the Linux arch-dispatch shim (POSIX `sh`).
- `scripts/pack-mcpb.sh` — download release assets (or `--from-dir`), lipo mac,
  lay out `server/bin/`, substitute version, `mcpb pack`, optional `--upload`.
  Requires a **macOS host** (lipo is macOS-only).
- `jake/mcpb.jake` — `mcpb.validate`, `mcpb.pack [tag=…]`, `mcpb.publish [tag=…]`.

## Bundle layout produced

```
manifest.json
server/bin/
  sema-linux              # POSIX shim → sema-linux-{x64,arm64} by uname -m
  sema-linux-x64
  sema-linux-arm64
  sema-macos-universal    # lipo(aarch64 + x86_64)
  sema-windows-x64.exe
```

## Local spike (verified 2026-07-17)

Built with 4 stand-in copies of the local `sema` (~32 MB each):
`mcpb validate` → pass; `mcpb pack` → `sema.mcpb`, 48 MB compressed / 122 MB
unpacked; exec bits (`rwxr-xr-x`) preserved inside the zip. Shim dispatch tested
against faked `uname -m`: `x86_64`→x64, `aarch64`→arm64, other→clean error.

**Size note:** the real bundle (universal mac ≈ 64 MB + 2 linux + win) is
~160 MB unpacked / ~60–70 MB compressed, and every user downloads all-OS
binaries. Acceptable for a single-asset install; revisit per-OS bundles only if
size becomes a complaint.

## CI wiring (do NOT edit release.yml — it's cargo-dist-autogenerated)

Add a separate workflow that fires after the release is published, on macOS
(for lipo), and packs+uploads:

```yaml
# .github/workflows/mcpb.yml
name: MCPB Bundle
on:
  release:
    types: [published]
permissions:
  contents: write            # upload the asset to the release
jobs:
  bundle:
    runs-on: macos-latest    # lipo is macOS-only
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: "22" }
      - name: Build + upload sema.mcpb
        env: { GH_TOKEN: "${{ github.token }}" }
        run: ./scripts/pack-mcpb.sh --tag "${{ github.event.release.tag_name }}" --upload
```

The `release: published` trigger means it also fires once cargo-dist finishes
creating the release, so the target-triple assets are present to download. (If a
race shows up, switch to `workflow_run` keyed on the Release workflow, or add a
short retry to the `gh release download` in the script.)

## Registry publish — DONE (2026-07-17)

**Live: `com.sema-lang/sema` v1.30.0, active.** Committed manifest:
`mcpb/server.json` (mcpb package → the v1.30.0 release asset, `fileSha256`
pinned, stdio transport).

Namespace note: `io.github.sema-lisp/*` was **not** grantable — the `sema-lisp`
org restricts third-party OAuth apps, so the registry's device-flow token can't
see admin membership and falls back to the personal namespace. We used the
DNS-verified **`com.sema-lang`** namespace instead (apex TXT on sema-lang.com,
added via Vercel), which sidesteps GitHub org OAuth entirely.

Re-publishing a new release:
1. Bump `version` + the asset URL/`fileSha256` in `mcpb/server.json`.
2. `mcp-publisher login dns --domain sema-lang.com --private-key <hex>` — needs
   the ed25519 key that matches the live apex TXT record. **That key is not in
   the repo.** Either preserve it (a GitHub Actions secret `MCP_PRIVATE_KEY` for
   an automated publish step in `mcpb.yml`), or regenerate a key + update the TXT
   record each time (the whole flow is scriptable).
3. `mcp-publisher publish` from `mcpb/`.
